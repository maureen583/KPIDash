# KPIDash

A sample manufacturing KPI dashboard built entirely with AI-assisted development
to demonstrate how modern development practices can leverage AI tooling without
ever exposing real client data. All operational data is synthetically generated —
no real proprietary information exists anywhere in this codebase or its container.

This project serves a dual purpose: it showcases what a real-time operations
dashboard could look like for a rubber mixing facility, and it proves that AI-assisted
development can be both powerful and privacy-safe by design.

The main purpose of this project is to demonstrate the privacy aspect of AI-Assisted development.
Some features are not fully production grade, to expedite the development process and allow for the
main goal of the project to be achieved.

---

## The Privacy Story

Every line of code in this project was written with Claude Code inside a locked-down
development container. The container's firewall restricts outbound traffic to a
strict allowlist — only GitHub, NuGet, npm, and Anthropic's API can be reached.
No client systems, no internal networks, no proprietary data sources. The SQLite
database is self-contained inside the container and seeded entirely with fake data
generated by the Bogus library.

---

## What It Simulates

A rubber mixing operation with four pieces of equipment running in sequence:

1. **Infeed Conveyor** — feeds raw rubber compound into the system
2. **Banbury Internal Mixer** — mixes rubber with carbon black, oils, and chemicals
3. **Mill** — sheets out the mixed compound for cooling
4. **Cooling Line** — brings compound down to handling temperature

Each piece of equipment has sensors that determine whether it is **Running**,
**Idle**, or **Down**. 7 days of synthetic sensor readings, batch records,
downtime events, and employee time logs are generated through running the seeder
project.

## Project Structure

### KPIDash.UI

Blazor WebAssembly frontend. Runs entirely in the browser and communicates with
the API exclusively over HTTP — it has no direct database access. Built with
MudBlazor for a clean, responsive dashboard experience. Contains pages for
equipment status, batch production, downtime tracking, and people utilization.

### KPIDash.API

ASP.NET Core Web API. The only layer with database access. Exposes endpoints
consumed by the UI and documents them automatically via Swagger at `/swagger`.
Also serves the compiled Blazor WASM files, meaning a single container runs
the entire application.

### KPIDash.Data

Class library that owns all database concerns. Uses Dapper for lightweight SQL
queries against a SQLite database. Manages schema via numbered migration scripts
that run automatically on startup — no manual database setup required. Repositories
are the only place raw SQL lives in the codebase.

### KPIDash.Services

Class library containing business logic. Sits between the API controllers and the
data repositories. Responsible for aggregating raw sensor readings into KPI metrics,
calculating utilization percentages, and shaping data into view models the UI can
consume directly.

### KPIDash.Seeder

Generates all synthetic data using the Bogus library. Produces 7 days of realistic
sensor readings,equipment status transitions, downtime events, batch records,
and employee shiftlogs. Data is internally consistent — downtime events align with
sensor anomalies,batches only occur when the Banbury is running, and shift
coverage matches production records.

## Running Locally

### Prerequisites

- Docker Desktop
- VS Code with the Dev Containers extension

### Steps

1. Clone the repository
2. Open in VS Code
3. When prompted, click **Reopen in Container**
4. Wait for the container to build (~3 minutes on first run)
5. In the terminal:

```bash
cd /workspace/KPIDash
dotnet run --project KPIDash.API
```

6. VS Code will prompt you to open the forwarded port in your browser
7. Navigate to `/swagger` to explore the API

The database is created and seeded automatically on first run. No setup required.

## Development Container

The devcontainer includes .NET 10 SDK, SQLite, and all required VS Code extensions.
A firewall script runs on container start that restricts outbound network access to
a strict allowlist. This is intentional — it mirrors the kind of network isolation
you would apply in a real client engagement to ensure the AI development tooling
cannot reach sensitive internal systems.
